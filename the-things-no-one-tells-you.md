# The Things That No One Tells You

## One Year Into My Coding Journey

A year ago, I started teaching myself how to code through a full-stack course on Udemy.

I began with HTML (which felt straightforward), then moved to CSS — where I quickly realised I was only scratching the surface. Two weeks in, I built my first website. Looking back, I barely understood things like properly targeting elements or structuring layouts.

Nine months later, I rewrote the entire site — not because it didn't work, but because I had improved that much.

Along the way I learned JavaScript, jQuery, EJS, PostgreSQL, React, Firebase and more. I built a working website for my business and even a fitness app using React and Firebase.

There were frustrating moments — studying everywhere (even in the rain), sleepless nights solving bugs, and times I felt like I was regressing.

But I wasn't regressing — I was trying to learn too much at once.

Now I'm focusing on mastering:
JavaScript, CSS, HTML, and Firebase.

Since narrowing my focus, I'm building faster, understanding deeper, and retaining more.

**Lesson learned:**
It's exciting to learn everything.
It's powerful to master something.

## How to Actually Learn to Code

Coding is the way of the modern age — but when you look up the different ways to learn, the first thing most people find are paid courses or a bootcamp. Let me break down what's actually worth your time and money.

### Paid Courses

You can easily pay for a course and learn what you need to know. Some providers offer packages that include full-stack training and a so-called "guaranteed job" at the end. Now yes, this sounds good — but in my opinion it's a scam. It's very expensive, and in the UK you don't need it. You could self-study and still land yourself a job with minimal to no experience at all.

### College Courses

Normal courses through colleges are a good starting point. You can do free starter courses to see what you take to at [FutureLearn](https://www.futurelearn.com) and even pay to do a Level 2 and 3 in coding. Worth considering if you want some structure and credentials.

### Bootcamps

Bootcamps are more for people that have the time. Some are shorter and more compressed but you'll get the knowledge needed. Some bootcamps are free but they run on a daily schedule with set sign-up dates — so if you work full time, this isn't really for you.

### YouTube

There are many people on YouTube you can learn coding from, and for a while this is how I was trying to learn. But honestly, it's a very bad way to learn. There's no structure, and all you do is double the time it takes to actually take in anything valuable.

### The Odin Project

One of the two best options for learning at your own pace without the pressure of deadlines. It's completely free — but it's all reading, no videos at all. [www.theodinproject.com](https://www.theodinproject.com)

### Udemy

This is what I used. I paid £12 for my course. Just to be clear — none of these courses are accredited. You're learning from a teacher on video and you're given a certificate of completion, not a degree. But Udemy is widely used by many people and companies, so the certificate still has some meaning. It could still help get you into an entry-level role.

## What I Wish Someone Had Told Me

The first thing I found out when learning to code is that you're basically learning *everything* — this is called full-stack. You learn front-end and back-end.

I went through video after video believing I needed to learn it all. I studied every day, brought my laptop everywhere — in a park, in the rain, wherever. I'd sit on a train going to work and write code in a notepad on my phone. I got to the point where I wasn't even sleeping most days. I'd study all night, then go to work on a couple of hours or no sleep at all.

I did this for over a year until I finally finished the course.

**And then I found out I never actually had to finish the course at all.**

What I needed was a plan from the start.

### Step 1: Know What You Want

Before you write a single line of code, ask yourself:
- Do I want to work for a company, or be my own boss?
- What kind of coding actually interests me — cybersecurity, websites, apps?

### Step 2: Learn the Foundations First

No matter what path you choose, you **100% need** to know: **HTML, CSS, and Git.** Full stop.

### Step 3: Pick Your Stack

Look up what is currently the most commonly used and popular. From my knowledge:
- **Front-end:** JavaScript and Python are highly popular. React is also great to know — it's a combination of HTML, CSS, and JS, but you need to understand those first before React makes sense.
- **Back-end:** I use Firebase. It's fairly simple compared to other backends — it handles authentication automatically, no hashing or schemas required.

### Timeline Expectations

**To land a job at a company:** You probably need a minimum of a year of study — fitting it around work. You'll need HTML, CSS, debugging, Git, JS or Python, some backend knowledge, and ideally React.

**To build something for yourself (using AI):** 6 months is realistic. You only need the basics — enough to spot a mistake AI makes and to think like a developer so you can instruct AI correctly. Because yes, AI can build pretty much anything you tell it to. But don't mistake that for thinking it works without your input and knowledge. There are always bugs, and it's on you to know how to spot them.

### AI and Coding Today

This day and age, whether you're working for yourself or at a company, you'll be using AI on a daily basis. When you're learning, use AI — but use it to a minimum. Don't just copy and paste. Make sure you understand the code first.

I found it useful to type everything up while learning. It helps you take things in a bit more.

# No-Fluff Web Dev Survival Guide

*For self-taught developers who are done watching and ready to build.*

---

## The Truth About Coding

You're not broken. You're not behind. You're overwhelmed — and that's by design.

The tutorial industry profits from your confusion. Courses are built to keep you watching, not shipping. You've probably started three courses, finished zero, and feel guilty about it. Stop.

Here's the truth: **you don't need to finish the course. You need to finish the project.**

Nobody learned to swim by watching videos. Nobody landed a dev job by completing every module. The developers getting hired — and the ones building real businesses — learned by breaking things, fixing them, and building again.

Burnout hits when you're consuming without creating. The fix isn't a better course. It's a narrower focus and a real project to aim at.

This guide skips the fluff. It gives you a 6-month roadmap, a debugging process, and the syntax you'll actually use. That's it. Now close the tutorial and open the editor.

---

## The 6-Month Roadmap

> **Core philosophy: Don't memorize, build. Don't finish the course, finish the project.**

| Month | Focus | Goal |
|-------|-------|------|
| **1** | HTML & CSS | Build a static personal site — real layout, real content. No templates. |
| **2** | JavaScript Fundamentals | DOM manipulation, events, fetch API. Add interactivity to your site. |
| **3** | JavaScript Deep Dive | ES6+, async/await, array methods, error handling. Break and fix things. |
| **4** | React | Components, props, state, hooks. Rebuild part of your site in React. |
| **5** | Firebase | Auth, Firestore, real-time data. Add login + live data to your React project. |
| **6** | Ship It | Polish your project, deploy it, and write about what you built. |

### Month-by-Month Breakdown

**Month 1 — HTML & CSS**
- Learn semantic HTML (header, main, section, article, footer)
- Learn the box model, flexbox, and basic responsive design
- **Project:** A personal site with at least 3 pages

**Month 2 — JavaScript Fundamentals**
- Variables, functions, loops, arrays, objects
- DOM selection and event listeners
- fetch() and basic API calls
- **Project:** Add a working contact form or live API widget to your site

**Month 3 — JavaScript Deep Dive**
- Arrow functions, destructuring, spread/rest operators
- Promises, async/await, try/catch
- Array methods: `.map()`, `.filter()`, `.reduce()`
- **Project:** A small JS app — todo list, weather checker, or quiz

**Month 4 — React**
- JSX, components, props, state with `useState`
- `useEffect` for side effects and data fetching
- Conditional rendering and list rendering
- **Project:** Rebuild your JS app as a React app

**Month 5 — Firebase**
- Firebase Auth (email/password and Google sign-in)
- Firestore: reading and writing documents
- Real-time listeners
- **Project:** Add user login and saved data to your React app

**Month 6 — Ship It**
- Deploy with Vercel or Netlify
- Clean up your GitHub and write a README
- Document one thing you built in a short blog post
- **Project:** Your finished, deployed, shareable app

## The Debugging Flowchart

When something breaks — and it will — follow this order. Don't skip steps.

```
Something is broken
        |
        v
1. CHECK THE CONSOLE
   ├── Is there a red error message?
   │   └── YES → Read it. Google the exact error. Fix it.
   └── NO → Move to step 2

2. CHECK YOUR LOGIC & DATA
   ├── Add console.log() before and after the broken area
   ├── Is the data what you expect? Is it undefined? Wrong type?
   │   └── YES (data is wrong) → Trace back where data comes from. Fix the source.
   └── NO (logic is wrong) → Walk through the code line by line. Rubber duck it.

3. CHECK THE UI / CSS
   ├── Is the element rendering but invisible or misplaced?
   ├── Open DevTools → Inspect Element
   │   └── Is a CSS rule overriding another? Is the element hidden?
   └── Fix the conflicting rule or add a border to debug layout

4. USE AI — BUT DO IT RIGHT
   ├── Don't paste your whole file and say "fix this"
   ├── DO say: "I'm using [tech]. I expected [X]. I got [Y]. Here's the relevant code: [snippet]"
   └── Read and understand the fix before applying it
```

**The golden rule:** If you can't explain what the bug is in one sentence, you haven't looked closely enough yet.

## The Essential Syntax Cheat Sheet

### Modern JavaScript (ES6+)

| Concept | Syntax |
|--------|--------|
| Arrow function | `const greet = (name) => \`Hello, ${name}\`` |
| Destructuring (object) | `const { name, age } = user` |
| Destructuring (array) | `const [first, second] = items` |
| Spread operator | `const newArr = [...arr, newItem]` |
| Optional chaining | `user?.profile?.avatar` |
| Nullish coalescing | `const name = user.name ?? 'Anonymous'` |
| Async/await | `const data = await fetch(url).then(r => r.json())` |
| Array map | `items.map(item => <li>{item}</li>)` |
| Array filter | `items.filter(item => item.active)` |
| Array find | `items.find(item => item.id === id)` |

---

### React Hooks

| Hook | When to use it | Example |
|------|---------------|---------|
| `useState` | Store and update local data | `const [count, setCount] = useState(0)` |
| `useEffect` | Run code on mount or when data changes | `useEffect(() => { fetchData() }, [id])` |
| `useRef` | Access a DOM element directly | `const inputRef = useRef(null)` |
| `useContext` | Share data across components without prop drilling | `const user = useContext(UserContext)` |

**useState pattern:**
```jsx
const [value, setValue] = useState('');

// Update it
setValue('new value');

// Update based on previous value
setValue(prev => prev + 1);
```

**useEffect pattern:**
```jsx
useEffect(() => {
  // Runs after render
  fetchData();

  return () => {
    // Cleanup (optional) — runs before next effect or unmount
  };
}, [dependency]); // Runs again when dependency changes
```

### Firebase Auth

| Task | Code |
|------|------|
| Sign up | `createUserWithEmailAndPassword(auth, email, password)` |
| Sign in | `signInWithEmailAndPassword(auth, email, password)` |
| Sign out | `signOut(auth)` |
| Google sign-in | `signInWithPopup(auth, new GoogleAuthProvider())` |
| Watch auth state | `onAuthStateChanged(auth, user => { ... })` |
| Get current user | `auth.currentUser` |

**Auth setup pattern:**
```js
import { getAuth, onAuthStateChanged } from 'firebase/auth';
const auth = getAuth();

onAuthStateChanged(auth, (user) => {
  if (user) {
    console.log('Logged in:', user.uid);
  } else {
    console.log('Logged out');
  }
});
```

### Firebase Firestore

| Task | Code |
|------|------|
| Add a document | `addDoc(collection(db, 'posts'), { title, body })` |
| Set a document (custom ID) | `setDoc(doc(db, 'users', uid), { name, email })` |
| Get a document | `getDoc(doc(db, 'users', uid))` |
| Get a collection | `getDocs(collection(db, 'posts'))` |
| Real-time listener | `onSnapshot(collection(db, 'posts'), snapshot => { ... })` |
| Update a field | `updateDoc(doc(db, 'users', uid), { name: 'New Name' })` |
| Delete a document | `deleteDoc(doc(db, 'users', uid))` |

**Read and display data pattern:**
```js
import { collection, onSnapshot } from 'firebase/firestore';

onSnapshot(collection(db, 'posts'), (snapshot) => {
  const posts = snapshot.docs.map(doc => ({
    id: doc.id,
    ...doc.data()
  }));
  setPosts(posts);
});
```
